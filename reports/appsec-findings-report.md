# AppSec Findings Report

This report summarizes the intentionally vulnerable training app in this lab. It is written to mirror the style of a real application security triage report.

| Finding ID | Tool | Finding Type | Severity | OWASP Category | Evidence | Business Risk | Recommended Fix | Owner | Status | Validation Step |
|---|---|---|---|---|---|---|---|---|---|---|
| AF-001 | Manual review / ZAP | Missing security headers | Medium | Security Misconfiguration | `/headers` reports hardened response security headers and an empty missing header list | Browser-based attacks and weak baseline hardening increase exposure for the public app | Implemented helmet baseline plus explicit Permissions-Policy header at app middleware layer | App developer | Closed | Verified `/headers` after remediation and captured `screenshots/after-security-headers.png` |
| AF-002 | Manual review | Debug endpoint exposed | High | Security Misconfiguration | `/debug` returns debug status and a fake secret string to anyone who can reach the app | Exposes internal details and demonstrates how accidental debug exposure can leak sensitive information | Remove the route in production or gate it behind authentication and environment controls | App developer | Open | Request `/debug` after the fix and confirm it returns 404 or a protected response |
| AF-003 | Manual review | IDOR-style broken access control | High | Broken Access Control | `/profile?id=1` returns profile data without any object-level authorization | A user may retrieve data for other users or internal roles if this were real identity data | Verify caller identity and enforce object-level authorization before returning profile data | App developer / API owner | Open | Attempt access with different IDs and confirm unauthorized access is blocked |
| AF-004 | Manual review | Excessive data exposure | High | Broken Access Control | `/api/orders/:id` now returns a minimized order view and excludes internal notes and payment token fragments | Over-shared API responses can leak internal data, support fraud, and increase privacy risk | Implemented response minimization so only required client fields are returned; next add object-level authorization controls | API owner | Closed | Verified `/api/orders/1001` excludes internal fields and captured `screenshots/after-api-data-exposure.png` |
| AF-005 | Gitleaks | Hardcoded secret pattern | High | Security Misconfiguration | `app/routes/debug.js` previously defined a hardcoded training secret that was surfaced by the `/debug` route | Demonstrates how real secrets in source can be harvested from code, logs, or images | Remove the hardcoded value, replace it with a non-secret placeholder, and rotate any real secret if one was ever used | Developer | Closed | Re-run `secrets-scan.yml` and confirm Gitleaks returns a clean pass with no secret pattern remaining |
| AF-006 | Dependabot / npm audit | Vulnerable dependency exposure | Medium | Vulnerable and Outdated Components | `app/package.json` uses third-party packages and is configured for Dependabot monitoring | Known package vulnerabilities can create exploit paths even when app code is clean | Review dependency alerts, upgrade packages, and re-test the app after update | App developer / build owner | Open | Re-run dependency scanning and confirm the alert is closed |
| AF-007 | Manual review | Unsafe input handling example | Medium | Injection | `/search?q=test` builds a simulated SQL string from raw input | Shows how injection risks arise when user input is concatenated into queries or commands | Validate and parameterize input instead of concatenating raw values | App developer | Open | Submit test payloads and confirm input is sanitized and no raw query string is built |

## Notes

- The app is intentionally vulnerable for training only.
- No real database, customer data, or secrets are used.
- The same triage flow used in vulnerability management still applies here: identify, validate, prioritize, coordinate, and re-test.
- AF-005 closure evidence: `Secrets Scan` run `28994165739` completed successfully after the hardcoded value was replaced with a placeholder.
- AF-001 remediation evidence: before/after captures completed (`screenshots/before-security-headers.png`, `screenshots/after-security-headers.png`) with header baseline validated.
- AF-004 remediation evidence: before/after captures completed (`screenshots/before-api-data-exposure.png`, `screenshots/after-api-data-exposure.png`) with minimized response validated.
